Detección y limpieza de servicios abandonados

Identificar contenedores sin uso real, dar backup verificable y dejarlos detenidos con período de gracia para eliminación final.

Fecha · 2026-05-10 Sistema · Docker · Dokploy · Vaultwarden

En un VPS donde se han probado muchas cosas a lo largo del tiempo es habitual que queden contenedores corriendo sin uso real, consumiendo RAM, CPU y superficie de ataque. El cliente tenía dos instancias de Vaultwarden simultáneas; una activa y la otra abandonada hace meses pero todavía aceptando tráfico de bots.

Cómo distinguir uno activo de uno abandonado

Un servicio "abandonado" no se reconoce por su uptime, sino por la última escritura real al estado y por la cantidad de tráfico legítimo vs. bots. Tres señales objetivas:

MétricaActivoAbandonado
Última escritura a la DBHoy / esta semana16 de enero (hace 4 meses)
Tamaño de DB2,5 MB244 KB (datos iniciales sin movimiento)
Tráfico recibido (90d)596 MB / 111 MB41 MB / 681 KB
Logs recientesRequests /api/config autenticadosSolo /admin/function.php (escaneos de bots)

Pasos seguros para dar de baja

El procedimiento "stop + cold backup" minimiza riesgos: el dato queda guardado en frío antes de cortar, y el contenedor se marca para no auto-reiniciarse en próximos reboots:

1. Cambiar la política de reinicio

docker update --restart=no <container_id>

Sin esto, un reboot del host volvería a levantar el contenedor por la RestartPolicy: always heredada del compose original.

2. Detener el contenedor en limpio

docker stop <container_id>

3. Cold backup del volumen

Identificar la ruta real del volumen Docker en el host:

docker inspect <container_id> \
  --format "{{range .Mounts}}{{.Source}} -> {{.Destination}}{{println}}{{end}}"

Copiar fríamente y empaquetar:

SRC=/var/lib/docker/volumes/<volume_name>/_data
DEST=/root/backups/vaultwarden-$(date +%Y-%m-%d)
mkdir -p "$DEST"
cp -a "$SRC"/. "$DEST"/
tar czf "$DEST.tgz" -C "$(dirname $DEST)" "$(basename $DEST)"

El backup queda como tarball además del directorio plano, listo para sftp o subir a almacenamiento frío.

Por qué no eliminamos en el momento

El cliente puede estar 90% seguro de que ese servicio ya no se usa, pero un 10% de duda existe. La política aplicada es: stop ahora, eliminar definitivamente en 2 semanas si nadie lo necesitó. La intervención agenda automáticamente un recordatorio para esa fecha que dispara una verificación remota y propone el cleanup final.

Si el cliente lo reactiva en esos 14 días, simplemente docker start. Si no, el comando final es:

docker rm <container_id>
docker volume rm <volume_name>
# Eliminar el stack del orquestador (Dokploy, en este caso)

Resultado

  • RAM liberada: ~15 MiB (poco para Vaultwarden, pero el patrón se aplica a stacks más pesados).
  • Superficie de ataque reducida: bot scans del puerto del subdominio abandonado ahora retornan 502 en vez de exponer la API de Vaultwarden.
  • Cero riesgo de pérdida: backup en disco + tarball, recuperable en cualquier momento del periodo de gracia.
  • Cierre formal automatizado: recordatorio agendado para confirmar eliminación final.
Aplicabilidad. Este patrón se aplica a cualquier servicio dudoso: paneles de prueba, instancias staging que nunca se borraron, builds antiguos de WordPress, herramientas internas que el equipo dejó de usar. La regla es siempre la misma: cold backup, stop, plazo de gracia, eliminación.
