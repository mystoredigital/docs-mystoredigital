Recuperación de Supavisor en Supabase self-hosted

INFRAESTRUCTURA 2026-05-10

Una instancia de Supabase self-hosted llevaba semanas con su pooler de conexiones Supavisor en crash loop sin que nadie lo notara. El contador de reintentos marcaba 9.896 al momento del diagnóstico. Las apps que dependían de pooling estaban funcionando intermitente o pasando por la conexión directa, perdiendo todo el beneficio del pool.

Sintoma

$ docker ps --filter "name=supavisor"
NAME                          STATUS
supabase-...-supavisor-1      Restarting (1) 45 seconds ago

$ docker inspect ... --format "{{.RestartCount}}"
9896

Diagnóstico

Los logs revelaron el error real, oculto bajo decenas de páginas de stack traces de Ecto:

** (Postgrex.Error) ERROR 3F000 (invalid_schema_name)
   schema "_supavisor" does not exist

Supavisor (escrito en Elixir, usa Ecto como ORM) intenta correr sus migraciones contra el schema _supavisor al arrancar. Si el schema no existe, la migración falla, el proceso muere, Docker reinicia el contenedor, y se repite. Es un bucle determinista que no se resuelve solo.

Causa raíz

Al provisionar Supabase self-hosted vía un orquestador (en este caso Dokploy), el script de inicialización del Postgres creó los schemas public, _supabase, _analytics, pero no _supavisor. Posiblemente por una versión del template anterior a que Supavisor fuera obligatorio.

Fix

Conectar a la base donde Supavisor está apuntando (la URL de conexión está en su variable de entorno DATABASE_URL), y crear el schema con el dueño correcto:

docker exec <db_container> psql -U supabase_admin -d _supabase \
  -c "CREATE SCHEMA IF NOT EXISTS _supavisor AUTHORIZATION supabase_admin;"

Tras crear el schema, el siguiente intento del crash loop encuentra el espacio donde correr migraciones, las aplica (crea las tablas internas que necesita), y arranca limpio. No hay que reiniciar el contenedor manualmente: Docker ya lo está reintentando cada pocos segundos.

Verificación

$ docker ps --filter "name=supavisor"
NAME                          STATUS
supabase-...-supavisor-1      Up 56 seconds (healthy)

$ docker logs --tail 5 supabase-...-supavisor-1
[info] HEAD /api/health
[info] Sent 204 in 369µs

Health checks fluyendo. Las apps que dependen del pooler recuperaron acceso transparentemente.

Lecciones

  • Los crash loops de Docker son silenciosos a nivel de monitoreo si no se observa específicamente RestartCount o se filtra docker ps por status. Vale la pena tener una alerta cuando algún contenedor supere N reintentos en M minutos.
  • Para diagnosticar errores de Ecto/Phoenix, ignora las primeras 30 líneas del stack: el error real suele estar bajo el ruido del migrator.
  • Los fixes de schema en bases productivas son idempotentes y baratos con CREATE SCHEMA IF NOT EXISTS. Si el schema ya existía, no pasa nada; si faltaba, queda creado.
Tiempo total de la intervención: ~4 minutos desde el diagnóstico hasta verificar healthy. El crash loop llevaba activo aproximadamente un mes.
