Acceso SSH gestionado y snapshot de seguridad

Provisioning automatizado de claves SSH al VPS y snapshot manual previo a cualquier intervención.

Fecha · 2026-05-10 Sistema · Hostinger API · Ubuntu 24.04

Antes de tocar producción se necesitan dos cosas: una vía de acceso con auditoría, y un punto de retorno si algo falla. Documentamos el patrón que aplicamos al VPS Hostinger del cliente, ejecutado de extremo a extremo desde la terminal del operador sin pasar por interfaces gráficas.

Paso 1 — Snapshot previo

Antes de modificar nada, se solicita un snapshot vía la API de Hostinger. Es instantáneo, gratuito en la mayoría de los planes y captura el disco completo del VPS:

POST /api/vps/v1/virtual-machines/{vm_id}/snapshot
Authorization: Bearer <HOSTINGER_API_TOKEN>

La respuesta devuelve un identificador de acción que se puede sondear para confirmar éxito. El snapshot expira a los 20 días (renovable). Cualquier cambio destructivo posterior se puede revertir con un solo POST a /snapshot/restore.

Por qué no nos saltamos este paso. El costo es cero, el tiempo de creación <3 segundos, y el de restauración ~30 minutos. Es la red de seguridad más barata que existe en infra.

Paso 2 — Inventario de claves existentes

Consultamos qué claves SSH están registradas en la cuenta y cuáles ya están adjuntas al VPS. Esto evita duplicar entradas y da contexto para auditar permisos:

GET /api/vps/v1/public-keys
GET /api/vps/v1/virtual-machines/{vm_id}/attached-public-keys

Paso 3 — Subida y vinculación de la clave

Generamos (o reusamos) un par de claves ed25519 en la máquina del operador, subimos la pública a Hostinger y la adjuntamos al VPS:

POST /api/vps/v1/public-keys
  body: { name: "Operator", key: "ssh-ed25519 AAAA..." }

POST /api/vps/v1/virtual-machines/{vm_id}/attach-public-keys
  body: { ids: [<public_key_id>] }
Gotcha conocido. Hostinger inyecta las claves vía cloud-init solo al crear o recrear el VM. Adjuntar una clave a un VM ya corriendo no la propaga al authorized_keys en vivo. La salida correcta cuando ya existen apps corriendo es bootstrap manual: pegar la clave una vez por consola web, y usarla a partir de ahí.

Paso 4 — Bootstrap del primer acceso

Para evitar reiniciar el VM, instalamos la clave directamente en el archivo del usuario destino, asegurando que la línea anterior termina en salto:

printf '\n%s\n' "ssh-ed25519 AAAA... operator@machine" >> ~/.ssh/authorized_keys
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys

Pequeño detalle pero importante: usar echo "..." >> sin verificar el final del archivo concatena la clave nueva al final de la línea previa cuando esa línea no termina en \n, dejando un authorized_keys sintácticamente inválido que sshd silenciosamente ignora. printf '\n%s\n' blinda contra eso.

Verificación

ssh -o BatchMode=yes root@<vps_ip> 'whoami; uname -a; uptime'

Si responde sin pedir password ni interacción, el acceso está operativo. A partir de aquí toda la administración se hace key-based, scriptable y auditable.

Resultado

  • Acceso SSH passwordless con clave ed25519, registrada y trazable en el panel de Hostinger.
  • Snapshot disponible como punto de retorno si algo sale mal.
  • Toda la operación tomó menos de 5 minutos vía API + un único comando manual de bootstrap.
