WhatsApp Agent — Roadmap de mejoras anti-ban

Análisis técnico del estado actual de las protecciones anti-bloqueo del agente y plan priorizado de mejoras para una versión robusta orientada a clientes enterprise o agencia que necesiten capacidades avanzadas (proxies por país, warm-up, dashboard de salud).

Fecha · 2026-05-27 Sistema · WhatsApp Agent · Baileys · Node 20 · OpenRouter Audiencia · Interna · Equipo técnico
Resumen

Qué hay hoy: el agente ya implementa 6 mecanismos anti-ban (rate limit per-chat + global, jitter humano 3-12s con presence "escribiendo", quiet hours 23-07, greylist contactos nuevos, reconexión exponencial, IA generativa que evita patrones). Cobre los vectores de riesgo del caso inbound.

Qué falta para una versión robusta: proxy opt-in (HTTP/SOCKS), warm-up gradual de números nuevos, detección y alerta de bloqueo, dashboard de salud del número, rotación outbound multi-número, spintax para snippets manuales.

Decisión vigente: no implementar red propia de proxies residenciales — costo prohibitivo ($5-20/GB) y baja ROI para el caso inbound-driven. Habilitar proxy como configuración opcional por número cuando un cliente lo requiera.

Contexto

Este documento responde a una pregunta recurrente que aparece cuando comparamos el agente con productos de mass-marketing tipo gogohl, que presumen "Red Global de Proxies 195+ países", "Modo Goteo Inteligente", "Retrasos Aleatorios" y "Soporte Spintax" como diferenciadores anti-ban.

La conclusión corta: varios de esos features ya los tenemos, otros no aplican a nuestro modelo, y unos pocos sí valen la pena implementar. Este doc detalla el porqué de cada uno y deja registro de qué construir cuando un cliente lo pague.

Capacidades anti-ban implementadas

Auditoría real del código (src/whatsapp.js) confirmó las siguientes protecciones activas:

MecanismoImplementaciónConfiguración por env
Rate limit per-chat Cooldown mínimo entre mensajes al mismo contacto. Bloquea respuestas rapid-fire. PER_CHAT_COOLDOWN_MS
Rate limit global Ventana móvil de 60s con cap configurable de mensajes salientes del agente. GLOBAL_MAX_PER_MIN (default 20)
Jitter humano Delay aleatorio antes de enviar (no fijo, distribución real con Math.random()). HUMAN_DELAY_MIN_MS · HUMAN_DELAY_MAX_MS (default 3-12s)
Composing presence Antes de enviar el agente emite sendPresenceUpdate('composing') — el contacto ve "escribiendo…" durante el delay. Visualmente indistinguible de un humano tipeando. —
Quiet hours Bloquea respuestas automáticas en franja nocturna (default 23:00–07:00 GYE). Las respuestas humanas manuales sí pasan. QUIET_HOURS_START · QUIET_HOURS_END · TIMEZONE
Greylist contactos nuevos El primer mensaje de un contacto desconocido siempre va a modo humano — la IA no responde para evitar auto-responder a spam que detonaría reportes. NEW_CONTACT_REQUIRES_HUMAN=true
Reconexión exponencial Si el socket Baileys cae, reintentos con backoff 3s → 6s → 12s … cap 60s. Evita martilleo que detonaría detección como bot. Hard-coded (3s base, 60s max)
IA generativa contextual Cada respuesta es única según el historial del contacto. No hay patrones de texto repetidos a múltiples contactos (a diferencia de plantillas masivas). OPENROUTER_API_KEY + system prompt editable

Modelo de riesgo: inbound vs outbound

La pregunta clave al diseñar features anti-ban: ¿qué actividad detona el ban? La respuesta cambia según el caso de uso, y el agente está construido para uno específico.

El agente es inbound-driven. El cliente escribe primero, la IA o el operador responde. Los volúmenes son moderados (cap 20 msg/min). No iniciamos conversaciones masivas con contactos fríos.
Vector de banProbabilidad inboundMitigación actual
Volumen masivo en burstBajaRate limit global 20/min
Latencia robot (respuesta <200ms)Alta sin mitigaciónJitter 3-12s + composing
Mensajes idénticos a muchos contactosNula con IAIA generativa contextual
Auto-responder a spamMediaGreylist + quiet hours
Iniciar conversaciones con contactos fríosNula (no es inbound)N/A — no aplica al caso
IP del VPS en blocklist de WhatsAppMuy bajaSin mitigación; agregar proxy opt-in cubre

Por contraste, en un producto de mass-marketing outbound (campañas a N contactos con plantilla), los vectores cambian: la IP empieza a importar mucho más (por eso la competencia presume 195+ países), la huella por número se vuelve crítica (de ahí el warm-up y la rotación), y el spintax se vuelve necesario para esquivar detección de plantillas repetidas.

Roadmap priorizado

Tres tiers basados en valor real para nuestro caso de uso vs costo de implementación.

Tier 1 — Alto valor / Bajo costo (implementar primero)

Próximos
  1. Detección y alerta de bloqueo — si el socket cae con code=401/403/loggedOut, emitir evento number.blocked por webhook + alert visible en UI + posible notificación Telegram/email. Hoy solo se loguea a consola. Tiempo: ~45min.
  2. Proxy opt-in vía env var — soportar WA_HTTP_PROXY o WA_SOCKS_PROXY por número, inyectado en socketConfig.agent de Baileys. Habilitar solo cuando un cliente lo necesite (IP corporativa, país específico, IP residencial). Tiempo: ~30min + costo del proxy externo (Bright Data, Smartproxy, etc.) pagado por el cliente. Tiempo: ~30min.
  3. Dashboard de salud del número — render en el panel actual con: uptime hoy, reconexiones esta semana, % rate-limit hits, último error. Las métricas ya existen en _limiter.snapshot(); falta el render UI. Tiempo: ~1h.
  4. Warm-up gradual de números nuevos — primer día cap 5 msg/h, segundo 10, tercero 20, hasta llegar al cap normal. WhatsApp es más estricto con números recién vinculados (<14 días). Tiempo: ~1h.

Tier 2 — Valor medio (cuando un cliente lo pida)

Bajo demanda
  1. Spintax para snippets manuales — sintaxis Hola {Juan|amigo|estimado} que randomiza al enviar. Solo para snippets/macros que el operador envía manual a múltiples contactos. NO para IA (ya es variada). Tiempo: ~1h.
  2. Rotación de número para outbound — si el tenant tiene N números, round-robin para contactos nuevos. Para contactos existentes ya hacemos routing por último número que habló. Tiempo: ~2h.
  3. Auto-pausa por fallos en cascada — 3 fallos consecutivos en 5min → desactivar IA del tenant + notificar. Protege contra ban en cascada. Tiempo: ~45min.
  4. Proxy automático por país del contacto — extender el proxy opt-in para que enrute por país detectado del prefijo telefónico (+57 Colombia → proxy CO). Solo si un cliente lo paga y justifica. Tiempo: ~3h + integración con proveedor de proxies.

Tier 3 — No recomendado replicar

Descartado
  • Red propia de proxies residenciales 195+ países. Costo $5-20/GB de tráfico residencial. No agrega valor para inbound (el cliente está contactándonos a él, la IP del VPS no importa). Si un cliente lo necesita, mejor consumir Bright Data / Smartproxy / Oxylabs bajo demanda y pasarle el costo en la factura.
  • Spintax obligatorio en IA. La IA ya genera variedad natural por contexto. Forzar spintax solo agrega ruido y reduce calidad de la respuesta.
  • Sistema propio de IP rotation por mensaje. Sobre-engineering. WhatsApp identifica el dispositivo por las llaves Signal del par auth, no solo por IP. Rotar IP por mensaje no ayuda y puede confundir a Baileys (re-handshake constante).

Bitácora de decisiones

2026-05-27 Mantener arquitectura inbound-driven sin proxy por default El caso de uso actual (responder a clientes que escriben) no se beneficia de proxies — la IP del VPS no es el vector de ban relevante. Agregar proxies por default sube complejidad y costo sin ROI claro.
2026-05-27 Habilitar proxy como configuración opt-in por número Para clientes con políticas de IP corporativa, requerimiento de IP local de un país, o que sufrieron ban del VPS, queremos poder activarlo sin re-arquitectar la app. Tier 1 del roadmap.
2026-05-27 No invertir en red propia de proxies residenciales Costo $5-20/GB residencial, baja ROI para inbound, mercado ya servido por Bright Data / Smartproxy. Si surge demanda real, integramos como proveedor externo.
2026-05-27 Priorizar alerta de bloqueo + dashboard de salud antes que proxy Hoy un número puede quedar desconectado y el operador se entera tarde. Visibilidad operativa > features nuevos que aún no se necesitan. Tier 1 #1 y #3.

Anexos técnicos

Anexo A — Snippet del RateLimiter actual

Ubicación: src/whatsapp.js · clase RateLimiter

class RateLimiter {
  constructor() {
    this._lastByJid = new Map();
    this._window = [];
  }
  check(jid) {
    const now = Date.now();
    this._window = this._window.filter((t) => now - t < 60_000);
    const last = this._lastByJid.get(jid) || 0;
    const sinceLast = now - last;
    if (sinceLast < PER_CHAT_COOLDOWN_MS) {
      return { ok: false, reason: 'per-chat cooldown', retryAfterMs: PER_CHAT_COOLDOWN_MS - sinceLast };
    }
    if (this._window.length >= GLOBAL_MAX_PER_MIN) {
      const oldest = this._window[0];
      return { ok: false, reason: 'global rate limit', retryAfterMs: Math.max(1000, 60_000 - (now - oldest)) };
    }
    return { ok: true };
  }
  record(jid) {
    const now = Date.now();
    this._lastByJid.set(jid, now);
    this._window.push(now);
  }
}

La clave del realismo: el delay incluye el tiempo de generación de IA. Si el modelo tarda 5s y el target del jitter cayó en 7s, esperamos 2s extra. Si tardó 13s y el target era 7s, enviamos inmediatamente. Comportamiento idéntico a un humano que "piensa antes de escribir".

Anexo C — Esqueleto del proxy opt-in (Tier 1 #2)

Snippet propuesto para implementar cuando se priorice. NO está en producción aún.

// En src/whatsapp.js — start()
import { HttpsProxyAgent } from 'https-proxy-agent';
import { SocksProxyAgent } from 'socks-proxy-agent';

const proxyUrl = this.numberConfig?.proxy
  || process.env[`WA_PROXY_${this.numberId.toUpperCase()}`]
  || process.env.WA_PROXY_DEFAULT;

const socketConfig = {
  version, auth: state, logger,
  printQRInTerminal: false,
  browser: ['Mac OS', 'Chrome', '120.0.0'],
};

if (proxyUrl) {
  const agent = proxyUrl.startsWith('socks')
    ? new SocksProxyAgent(proxyUrl)
    : new HttpsProxyAgent(proxyUrl);
  socketConfig.agent = agent;
  socketConfig.fetchAgent = agent;
  console.log(`[wa:${this._tag}] usando proxy ${new URL(proxyUrl).host}`);
}

this.sock = makeWASocket(socketConfig);

Dependencias: https-proxy-agent (ya transitiva via Baileys) y socks-proxy-agent (a agregar al package.json si se quiere SOCKS5).

Cierre

El agente cubre los vectores de ban relevantes para su caso de uso (inbound-driven, volúmenes moderados). El gap real frente a la competencia outbound-masivo es honesto: no tenemos proxies por país ni warm-up automático. Ninguno de los dos es crítico hoy; ambos son implementables en horas cuando un cliente lo pague.

El próximo paso natural cuando aparezca un cliente que requiera "versión robusta" es Tier 1 completo (alerta de bloqueo + proxy opt-in + dashboard de salud + warm-up) en ~3h de trabajo. Eso da paridad funcional con la competencia para el 95% de los casos reales, sin invertir en infraestructura que no se justifica.

Posicionamiento defendible. Frente a un cliente que compara: nuestro agente es contextualmente más inteligente (IA generativa por conversación), operativamente más seguro (greylist + quiet hours + composing realista), y configurable a profundidad (proxy bajo demanda, rate limits por env, multi-tenant). No competimos en cantidad de IPs porque ese no es el problema que resuelve nuestro caso de uso.
