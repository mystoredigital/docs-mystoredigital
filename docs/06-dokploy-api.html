API de Dokploy para despliegues programáticos

Auth con x-api-key (no Bearer), tRPC procedures bajo /api/<router>.<procedure>, write seguro vía throwaway test.

Fecha · 2026-05-10 Sistema · Dokploy v0.29.x · tRPC

Dokploy expone un API HTTP funcional, pero su documentación oficial es escasa y el patrón de auth no sigue el estándar habitual. Documentamos aquí lo que descubrimos para que el siguiente despliegue sea inmediato.

Autenticación: x-api-key, no Authorization: Bearer

El token se obtiene desde el panel web (Profile → API Keys) y se envía en el header x-api-key. Si se usa Authorization: Bearer <token>, la API responde 401 Unauthorized independientemente de si el token es válido. Es el primer descubrimiento crítico para no perder tiempo.

curl -H "x-api-key: $DOKPLOY_API_TOKEN" https://<panel>/api/project.all
[{"projectId":"...","name":"Apps", ...}]

Estilo del API: tRPC procedures

Dokploy usa tRPC bajo el capó. Los endpoints siguen el patrón /api/<router>.<procedure>, no /api/v1/resource. Algunos ejemplos relevantes:

EndpointMétodoUso
project.allGETListar proyectos visibles para el token
project.one?projectId=...GETDetalle del proyecto + environments + apps
project.createPOSTCrear proyecto. Body: {name, description}
project.removePOSTEliminar. Body: {projectId}
compose.createPOSTCrear stack docker-compose
compose.updatePOSTCambiar provider, repo, branch, etc.
compose.deployPOSTForzar deploy
domain.createPOSTRegistrar dominio asociado a compose/app
gitProvider.getAllGETListar providers Git conectados (para githubId)

La especificación OpenAPI completa está en /api/openapi.json. Es la fuente de verdad cuando se necesita un endpoint que no aparezca aquí.

Modelo de datos

La jerarquía es: Organization → Project → Environment → Application | Compose. Una organización suele ser una cuenta de cliente; los proyectos agrupan apps relacionadas; cada proyecto tiene al menos un environment (production por default); las apps viven dentro del environment.

Un token está scoped a una organización. Si el cliente tiene varios equipos en diferentes organizations, se necesita un token por organization.

Test de write con un throwaway

Antes de confiar en que un token tiene permisos efectivos, conviene un test idempotente:

# Crear
POST /api/project.create
  body: {"name":"smoke-test","description":"throwaway"}
  → response: { project: { projectId: "..." }, environment: {...} }

# Eliminar
POST /api/project.remove
  body: {"projectId":"..."}
  → response: project eliminado

# Confirmar
GET /api/project.all
  → smoke-test ya no aparece

30 segundos en total, confirma read + write sin riesgo de tocar nada productivo.

Despliegue desde repo Git (workflow recomendado)

El patrón productivo es: el código vive en GitHub, Dokploy se conecta vía GitHub App, y a cada push hace pull + rebuild + redeploy. La creación del app vía API:

POST /api/compose.create
  body: {
    name: "docs",
    environmentId: "<production_env_id>"
  }

# Luego completar con el provider
POST /api/compose.update
  body: {
    composeId: "...",
    sourceType: "github",
    githubId: "<de gitProvider.getAll>",
    owner: "My-Store-Digital-Team",
    repository: "docs-mystoredigital",
    branch: "main",
    composePath: "./docker-compose.yml",
    autoDeploy: true
  }

# Disparar primer deploy
POST /api/compose.deploy
  body: { composeId: "..." }

Una vez creado, push a main dispara el redeploy automáticamente si el GitHub App tiene acceso al repo (instalado en la org o usuario propietario).

Detalle importante: el GitHub App de Dokploy debe estar instalado en el mismo namespace (usuario u organización) donde vive el repo. Si está instalado solo en el usuario personal y el repo está en la org, los webhooks no se entregan y el auto-deploy no funciona. Solución: visitar https://github.com/apps/<app-slug>/installations/new e instalar en la org adicional.

Cuándo usar API y cuándo no

  • Sí: creación masiva de apps de un cliente, automatización de provisioning post-onboarding, scripts de migración entre proyectos, integraciones con CRM (cliente firma → app provisionada).
  • No (todavía): ediciones puntuales de variables de entorno o re-config UI, donde el panel web es más rápido y menos propenso a errores.
Alternativa: si el cliente no quiere depender del panel de Dokploy y prefiere control puro, todo el stack se puede desplegar fuera de Dokploy con docker compose up -d directo, conectándose al mismo network dokploy-network que usa el Traefik existente. El reverse proxy y los certificados siguen funcionando idénticos. La pérdida es solo la visualización en el dashboard.
