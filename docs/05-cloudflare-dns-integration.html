Automatización

Integración programática con Cloudflare DNS

Token con scope mínimo y gestión idempotente de registros A para subdominios de apps, sin tocar la UI.

Fecha · 2026-05-10 Sistema · Cloudflare API v4 · Let's Encrypt

Para que un despliegue sea verdaderamente end-to-end, el subdominio donde la app va a vivir tiene que crearse automáticamente. Cloudflare expone una API REST limpia que permite hacerlo en una llamada, una vez se tiene un token con scope correcto.

Generación del token con scope mínimo

El principio aplicado es least privilege: el token solo puede editar registros DNS de los zones explícitos que va a gestionar, no toca firewall, no toca billing, no puede crear nuevos zones.

En https://dash.cloudflare.com/profile/api-tokens se crea un token con plantilla "Edit zone DNS" o configuración custom:

  • Permission: Zone — DNS — Edit
  • Permission: Zone — Zone — Read
  • Zone Resources: específicos (solo los dominios bajo gestión, no "all zones")

Verificación del token y listado de zones accesibles:

curl -H "Authorization: Bearer $CLOUDFLARE_API_TOKEN" \
  https://api.cloudflare.com/client/v4/user/tokens/verify

curl -H "Authorization: Bearer $CLOUDFLARE_API_TOKEN" \
  "https://api.cloudflare.com/client/v4/zones?per_page=50"

Esquema de operación típico

1. Resolver el zone_id del dominio

GET /zones?name=mystoredigital.cloud
→ result[0].id  →  a29f55cabbff8a3262bb6bc1e4d4d0ee

El zone_id es estable; en deployments repetidos se cachea para evitar la llamada extra.

2. Buscar si ya existe el registro

GET /zones/{zone_id}/dns_records?name=docs.mystoredigital.cloud&type=A

3. Crear o actualizar (idempotente)

POST /zones/{zone_id}/dns_records
{
  "type": "A",
  "name": "docs.mystoredigital.cloud",
  "content": "72.60.27.13",
  "ttl": 1,
  "proxied": true,
  "comment": "Auto-creado por pipeline de despliegue"
}

ttl: 1 equivale a "automático" en Cloudflare. proxied: true activa la nube naranja: el tráfico va por edge de Cloudflare con CDN, DDoS protection y HTTPS terminación en edge. proxied: false deja DNS-only para servicios donde se necesita la IP real (SSH, conexiones TCP no-HTTP).

Convivencia con Let's Encrypt en el origen

Una pregunta frecuente: si Cloudflare ya termina HTTPS en su edge, ¿necesitamos certificado en el origen? Sí, si el modo SSL del zone está en Full o Full (Strict), Cloudflare necesita conectarse al origen vía HTTPS. Traefik resuelve esto con Let's Encrypt automáticamente, vía challenge HTTP-01 que pasa transparente a través del proxy de Cloudflare.

Gotcha. El challenge HTTP-01 funciona porque Cloudflare proxy permite el tráfico hacia /.well-known/acme-challenge/... sin alterarlo. Pero si tienes habilitado Always Use HTTPS con HSTS agresivo o Page Rules que reescriben URLs, el challenge puede fallar. La salida segura es el challenge DNS-01 (que requiere darle al certificador permisos de DNS también).

Lo que esto destraba

Combinado con el resto del pipeline (GitHub + Dokploy + Traefik), un despliegue completo se reduce a:

# 1. Push de código
git push

# 2. Crear DNS automáticamente
POST /zones/{id}/dns_records  (proxied=true → 72.60.27.13)

# 3. Dokploy detecta el push, rebuild, redeploy
# 4. Traefik solicita cert al edge → Let's Encrypt
# 5. Cloudflare termina HTTPS en edge con su propio cert universal

→ https://nuevaapp.tudominio.com  (SSL + CDN + DDoS protect)

Tiempo total típico desde git push hasta URL operativa: 2-4 minutos, dominado por el build de Docker y la propagación DNS (que Cloudflare hace cuasi-instantánea).

Auditoría de cambios

Cada llamada de la API pasa por logs en Audit Logs de Cloudflare, identificada por el token. Si en algún momento se compromete o se sospecha de un cambio rogue, se revoca el token desde el dashboard y se rota; los registros que el token creó siguen intactos.
