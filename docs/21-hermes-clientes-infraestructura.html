Hermes para Clientes — Infraestructura

Un agente Hermes aislado por cliente en el homelab Proxmox, accesible de forma remota con login y operable por Telegram 24/7.

Fecha · 2026-06-04 Sistema · Proxmox + Hermes Agent + Cloudflare (Tunnel/Access) + Telegram
Resumen

Qué se hizo: se montó un prototipo donde cada cliente tiene su propio agente Hermes en un contenedor LXC, accesible desde internet por un panel web protegido con login.

Cómo: se clona el Hermes base (LXC 108), se expone con un túnel de Cloudflare + Cloudflare Access (login por email), y el uso diario se hace por Telegram (sin abrir puertos del router).

Resultado: primer cliente funcionando en hermes-cliente1.mystoredigital.cloud. Hermes es muy liviano (~0.4 GB RAM en reposo), así que caben ~20 clientes en el nodo actual.

Qué es Hermes y qué resuelve

Hermes es un agente de IA open source (Nous Research) que no solo conversa, sino que ejecuta acciones: se conecta a herramientas, corre tareas programadas y opera 24/7. La idea de este proyecto es ofrecer a cada cliente su propia instancia aislada, que pueda usar de forma remota.

El reto principal no es Hermes en sí, sino que el servidor vive en una red doméstica (el homelab Proxmox). Exponerlo a internet de forma segura se resolvió con Cloudflare Tunnel + Access, sin abrir puertos del router ni exponer la IP de casa.

OK. Un cliente cabe holgado en 1 vCPU / 1.5 GB / 10 GB. En el nodo de 31 GB de RAM caben aproximadamente 20 clientes. El disco no es limitante (thin provisioning, ~600 GB libres).

Arquitectura

Dos vías de acceso, complementarias: panel web (para administrar/configurar) y Telegram (para el uso diario).

Panel web:
Internet → Cloudflare (HTTPS) → Cloudflare Access (login por email)
        → Cloudflare Tunnel (cloudflared en el LXC) → Hermes Dashboard :9119

Telegram:
Cliente → Bot de Telegram → (long polling, saliente) → hermes-gateway (LXC)
Info. Telegram no necesita túnel ni Cloudflare: el gateway se conecta hacia Telegram (saliente), así que funciona directo desde la red doméstica. Por eso es la vía recomendada para el día a día.

Receta: añadir un cliente nuevo

  1. Clonar el LXC 108 desde el snapshot limpio clean-install
  2. Asignar recursos (1 vCPU / 1.5 GB / 10 GB) y arrancar
  3. Poner la clave LLM del cliente en /root/.hermes/.env
  4. Levantar el panel web hermes-dashboard (puerto 9119)
  5. Crear túnel Cloudflare + DNS + ingress a 127.0.0.1:9119
  6. Crear app de Cloudflare Access con el email del cliente
  7. Conectar Telegram (bot + autorizar usuario) para uso diario
Detalle técnico — gotcha del Host header resuelto

El dashboard se ata a 127.0.0.1 y valida el header Host. Al llegar la petición por el túnel con el hostname público, la rechazaba con "Invalid Host header". Se resolvió diciéndole al túnel que reescriba el Host:

ingress → originRequest.httpHostHeader = "127.0.0.1:9119"

Conectar Telegram

Para operar el agente por Telegram (recomendado), el cliente solo necesita:

  1. Crear un bot con @BotFather (/newbot) → obtener el bot token.
  2. Obtener su Id de Telegram (con @userinfobot) para autorizarlo.

Con esos dos datos se configura TELEGRAM_BOT_TOKEN + TELEGRAM_ALLOWED_USERS en el clon y se arranca el servicio hermes-gateway. A partir de ahí, el cliente le escribe al bot y el agente responde 24/7.

Pendientes antes de cliente real

Advertencia. El prototipo es funcional, pero antes de entregarlo a un cliente que pague hay que cerrar estos puntos.
  1. Cambiar la clave OpenRouter temporal por la del cliente (o una gestionada con tope de gasto)
  2. El panel muestra la pestaña de claves (API keys) tras el login → restringir o usar vista solo-chat
  3. Cambiar el email autorizado en Access por el del cliente
  4. Definir el modelo de cobro del LLM
  5. (Opcional) Login con Google en vez de PIN por email — más fiable, 1 clic
Resultado. Base reproducible para un add-on comercial: cada cliente con su agente Hermes aislado, seguro y operable por Telegram. Añadir un cliente nuevo es repetir la receta (automatizable).
