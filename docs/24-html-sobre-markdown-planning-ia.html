HTML sobre Markdown para planning con IA

Por qué los planes de tareas que un humano va a auditar deben ser HTML, no Markdown — y cómo lo aplicamos en el flujo de trabajo con Claude Code.

Fecha · 2026-06-10 Sistema · Claude Code · HTML · metodología human-in-the-loop
Resumen

Qué se propone: usar HTML self-contained (no Markdown) para los planes de tareas no triviales que un humano va a revisar antes de ejecutar.

Por qué: el lift de rendimiento del modelo es despreciable; lo que cambia es la auditabilidad. El formato HTML obliga a mirar en serio, en lugar de escanear un plan en diagonal y darle el visto bueno por inercia.

Resultado: separamos captura (sigue en Markdown) de planning ejecutable (pasa a HTML auditable), reduciendo el riesgo de descubrir a mitad de camino que una fase tenía una dependencia mal puesta.

El problema: leer no es auditar

Cuando un agente como Claude Code genera un plan en Markdown, lo normal es escanearlo en diagonal: ves los encabezados de fase, te convences de que "se ve bien" y sigues. Eso no es auditar — es confiar a ciegas. El plan parece sólido hasta que, ya ejecutando, descubres que la fase 2 nunca tuvo la dependencia correcta.

La tesis que origina esta metodología es directa: para planes que un humano va a leer (no solo el modelo), HTML supera a Markdown. La mejora en el rendimiento del propio modelo es marginal. Lo que realmente cambia es la capacidad del humano de revisar el plan con rigor.

Origen. La idea proviene de una observación de Anthropic sobre la "efectividad poco razonable del HTML" para planes legibles por humanos, popularizada por Mark Kashef (Early AI Dopters). Aquí está destilada y adaptada al flujo de trabajo del estudio.

Por qué el HTML resuelve el problema

El punto no es estético. Un plan en HTML cambia el comportamiento del lector porque la estructura visual obliga a detenerse en cada parte en lugar de deslizarse por una pared de texto monocromática.

Plan en MarkdownPlan en HTML
Lectura en diagonal — todo "se ve ok"Lectura forzada por la estructura visual
Encabezados planos ## ###Tabs, secciones colapsables, paneles
El ASCII art se rompe entre fuentes proporcionales<pre> mantiene el monoespaciado intacto
Estado de fase = emoji o texto sueltoEstado = color + etiqueta + timestamp
Una decisión cambia → editas y pierdes el hiloUna decisión cambia → entrada nueva con timestamp
Confías a ciegasAuditas porque el formato te obliga a mirar

Las ventajas concretas

  1. Claridad visual — fácil de recorrer con la vista; no es un bloque uniforme de texto.
  2. Tabs y secciones plegables — entras a una fase específica sin que el resto te tape la vista.
  3. El ASCII art funciona — los diagramas de arquitectura se siguen viendo bien dentro de <pre>.
  4. Edición en vivo — el agente puede actualizar el HTML mientras trabaja y registrar decisiones en tiempo real, cambiando de rumbo cuando hace falta, en lugar de solo "avisar" en el chat.
Advertencia. Esto no es "todo a HTML". Para captura, notas, ideas y journaling, Markdown sigue siendo lo correcto: rápido, grep-able, plain text. El HTML entra solo cuando el output necesita auditoría humana antes de ejecutarse.

El cambio de flujo

Antes — confianza por inercia

Frágil
  1. El agente genera el plan en Markdown.
  2. Lo lees en Obsidian.
  3. Asumes que está bien.
  4. Ejecutas.
  5. A mitad de camino descubres que la fase 2 no tenía la dependencia correcta.

Después — auditoría forzada

Robusto
  1. Captura, notas e ideas siguen en Markdown. No se toca.
  2. Para planes de tareas no triviales, le pides al agente un HTML self-contained.
  3. Guardas los planes en una carpeta separada (p. ej. _planes/).
  4. Abres cada plan en el navegador antes de ejecutar. El formato obliga la auditoría.
  5. El agente actualiza el HTML mientras trabaja — no solo te avisa en el chat.

El prompt para generarlo

El plan se pide explícitamente como un único archivo HTML autocontenido, sin librerías externas, para poder abrirlo en cualquier navegador sin dependencias:

Antes de tocar código, dame el plan completo como UN archivo HTML
self-contained (CSS inline, sin librerías externas) con:

- Header: nombre del proyecto + objetivo en una línea + riesgos
- Una sección por fase, colapsable con <details>
- Estado por fase (pending / in_progress / done) con color de fondo
- Sub-pasos numerados dentro de cada fase
- Panel lateral "Decisiones" con timestamp + razón
- Diagrama ASCII en <pre> si hay arquitectura
- Botón final "Aprobar y ejecutar"

Output: un solo .html que pueda abrir en el navegador.
NO lo abras tú, solo dame el archivo.
Patrón de checkpoint. Un uso real adicional: cuando una sesión de trabajo larga llena el context window, el plan HTML sirve como punto de control. En lugar de copiar y pegar para reiniciar, retomas desde un documento que ya refleja el estado y las decisiones tomadas.

Conexión con el resto del estudio

Esta práctica generaliza un patrón que ya usamos:

Dashboards HTML
Mission Control y otros paneles ya usan HTML como salida. Aquí extendemos el patrón: todo plan debería ser HTML auditable, no solo el dashboard final.
Obsidian
Renderiza HTML embebido, así que un .html puede vivir directamente en el vault sin convertir nada.
docs.mystoredigital
La idea de "documento HTML auditable" ya vive aquí. Lo nuevo es aplicarla internamente al planning, no solo al entregable final del cliente.

La regla que adoptamos

2026-06-10 Separar captura de planning ejecutable. Markdown es óptimo para input rápido y grep-able; HTML es óptimo para output que necesita auditoría antes de actuar. Son dos trabajos distintos.
2026-06-10 Captura, journaling, notas y referencias → Markdown en Obsidian. Velocidad y portabilidad en texto plano. No requieren auditoría visual.
2026-06-10 Planes de tareas que se ejecutan con Claude Code → HTML self-contained. El formato obliga a revisar de verdad y permite registrar decisiones con timestamp mientras el agente trabaja.
En una línea. Markdown se queda para el input rápido. HTML entra para el output que necesita auditoría.
