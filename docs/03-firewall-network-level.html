Firewall network-level con default-deny

Whitelist explícita a nivel de hipervisor. Reducción de 7 a 4 puertos expuestos, sin downtime, sin conflictos con Docker.

Fecha · 2026-05-10 Sistema · Hostinger Firewall · Traefik · Docker Swarm

Un VPS típico provisionado de fábrica deja todos los puertos abiertos al mundo. Cuando ya tiene apps corriendo, eso significa exponer dashboards internos, panels de admin, bases de datos directas y cualquier puerto que un servicio publique por error. La solución correcta es un firewall a nivel de hipervisor con default-deny, no UFW dentro del VM.

Por qué firewall de hipervisor y no UFW

  • Docker bypassa iptables. Cuando Docker publica un puerto vía -p 80:80, lo agrega a una cadena propia (DOCKER) que se ejecuta antes que las reglas de UFW. Resultado: tu UFW puede decir DENY en el 80 y el contenedor sigue siendo accesible.
  • Hypervisor firewall corre antes del VM. El paquete jamás llega al VM si la regla lo bloquea. Cero latencia añadida y cero conflicto con orquestadores.
  • Cambios sin downtime. Las reglas se aplican en caliente, sin reiniciar nada.

Inventario previo: qué puertos están escuchando

$ ss -tlnp | head -10
LISTEN  0.0.0.0:443      docker-proxy   # Traefik
LISTEN  0.0.0.0:80       docker-proxy   # Traefik
LISTEN  0.0.0.0:32769    docker-proxy   # Nextcloud directo
LISTEN  0.0.0.0:22       sshd
LISTEN  0.0.0.0:3000     docker-proxy   # Dokploy panel directo
LISTEN  0.0.0.0:31000    docker-proxy   # MDM device check-in

Tres tipos de puertos:

  1. Necesarios al mundo: 80, 443 (Traefik) y posiblemente uno específico de protocolo (en este caso 31000 para clientes MDM).
  2. Necesarios pero solo a tu IP: 22 (SSH).
  3. Innecesarios al mundo: 3000 y 32769 — son interfaces de admin que ya están detrás de Traefik con un subdominio HTTPS. Exponerlas directo es duplicar la superficie sin agregar valor.

Reglas aplicadas

ProtocoloPuertoOrigenJustificación
TCP22IP del operadorSSH limitado, fail2ban innecesario
TCP80anyTraefik HTTP (redirige a HTTPS)
TCP443anyTraefik HTTPS
UDP443anyHTTP/3 (QUIC) vía Traefik
TCP31000anyMDM device check-in
ICMP-anyPing/traceroute para monitoreo

Todo lo demás cae al default-deny. Los puertos 3000 y 32769 ya no responden desde fuera, pero siguen accesibles internamente y por su subdominio HTTPS detrás de Traefik.

Implementación vía API

POST /api/vps/v1/firewalls
  body: { name: "claudeclaw-vps" }

# Por cada regla:
POST /api/vps/v1/firewalls/{id}/rules
  body: {
    protocol: "TCP",
    port: "22",
    source: "custom",
    source_detail: "<ip_operador>"
  }

# Activar el firewall en el VM:
POST /api/vps/v1/virtual-machines/{vm_id}/firewall/{firewall_id}

Verificación

Desde la IP del operador:

$ ssh root@vps "echo OK"
OK                              # SSH permitido para mi IP

$ curl -m 5 http://<vps_ip>:443
HTTP 404                        # responde, esperado

$ curl -m 5 http://<vps_ip>:3000
(timeout)                       # bloqueado, esperado

Notas operativas

  • Si la IP del operador cambia (red móvil, viaje), se actualiza la regla por API en 5 segundos.
  • El default-deny aplica también a los puertos del Docker Swarm (2377, 7946, 4789), que estaban escuchando en * pero solo deberían usarse entre nodos del cluster — quedaron correctamente cerrados al exterior.
  • Conexiones ya establecidas no se cortan al activar el firewall, así que se puede aplicar sin downtime.
Reducción de superficie: de 7 puertos expuestos al mundo a 4. Los dos paneles internos (Dokploy y Nextcloud direct port) ya no son alcanzables desde Internet, eliminando vectores de ataque dirigidos a versiones específicas de esos servicios.
