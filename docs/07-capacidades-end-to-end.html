Del prompt a la URL pública en minutos

El pipeline completo: código generado, Cloudflare DNS, repo en GitHub, Dokploy + Traefik, SSL automático. Todo encadenado.

Fecha · 2026-05-10 Sistema · GitHub · Cloudflare · Dokploy · Traefik · Let's Encrypt

Las integraciones documentadas en las otras entradas componen un pipeline en el que un cliente describe lo que necesita y, sin tocar consolas, dashboards ni archivos de configuración manualmente, termina con una URL pública servida con SSL detrás de un CDN. Esta entrada describe el flujo completo y lo que se requiere para reproducirlo.

El flujo, paso por paso

Input: "Quiero que la herramienta interna del equipo de operaciones esté disponible en ops.cliente.com".

1. Generación del código

Si la app no existe, se genera el código (frontend, backend, schema de DB) en un repo nuevo. Si ya existe, se clona y adapta. Lo que se entrega siempre incluye: Dockerfile productivo, docker-compose.yml con labels Traefik, configuración de variables de entorno con valores razonables.

2. Push a GitHub

El repo se crea en la organización del cliente, con visibilidad y branch protection según política. El operador se autentica con un PAT pre-configurado.

3. Creación del registro DNS

Una llamada al API de Cloudflare resuelve el zone, busca si el subdominio ya existe (idempotencia), y crea o actualiza el registro A apuntando a la IP del VPS. Por defecto se activa el proxy naranja (CDN + DDoS protection).

4. Provisioning en el orquestador

Una llamada al API de Dokploy crea el stack, lo conecta al repo, y dispara el primer build. Alternativamente, si se prefiere fuera del panel: SSH al VPS, git clone, docker compose up -d. Ambos caminos terminan en el mismo Traefik con los mismos certificados.

5. Build y despliegue

Docker hace el build multi-stage (Node/Bun para frontend, Go/Python/Node para backend, según el caso), aplica las variables de entorno, levanta los contenedores en la red dokploy-network.

6. Negociación de SSL

Traefik detecta el nuevo router gracias a las labels, solicita certificado a Let's Encrypt vía challenge HTTP-01, y empieza a servir HTTPS en cuanto el cert se emite (típicamente <30 segundos).

7. Verificación automática

El pipeline hace curl https://ops.cliente.com y espera 200. Si falla, dispara un fallback al operador con los logs relevantes. Si pasa, se notifica completion.

Lo que un cliente nuevo necesita aportar

  1. Acceso al VPS (Hostinger u otro). Si es nuevo, se provisiona en 5 min con la API.
  2. Dominio gestionado en Cloudflare (recomendado) o cualquier DNS donde se pueda automatizar. Necesita un token API con scope a los zones donde se vayan a crear subdominios.
  3. Cuenta de GitHub con organización para los repos.
  4. Email para Let's Encrypt. Cualquier dirección válida; se usa para alertas de expiración.

Con esos cuatro inputs configurados una sola vez, cada despliegue posterior no requiere nada nuevo del cliente.

Lo que queda fuera (intencionalmente)

Algunos pasos NO se automatizan a propósito porque requieren juicio o son destructivos:

  • Eliminación de servicios. Siempre con periodo de gracia y backup explícito (ver cleanup de Vaultwarden).
  • Migraciones de schema en producción con riesgo de pérdida de datos.
  • Cambios al firewall que afecten SSH del propio operador sin verificación previa de la regla.
  • Ediciones en el DNS de zones críticos (registros raíz, MX, SPF, DKIM) sin confirmación humana.

Para todos esos, el patrón es: proponer la acción, mostrar el diff, esperar confirmación. La velocidad se gana en lo seguro y reversible; la cautela se reserva para lo destructivo.

Por qué importa este flujo

Tradicionalmente, lanzar una app interna o una landing nueva del cliente cruza por: desarrollador genera código, devops configura servidor, alguien tramita el dominio, alguien más solicita el cert, alguien actualiza DNS, alguien valida que esté vivo. Cinco roles, varios días, y cada handoff es una oportunidad de bloqueo.

Con el pipeline descrito, esa cadena la ejecuta una sola persona (o agente), de forma trazable, en minutos en lugar de días. La capacidad técnica que tradicionalmente requería un equipo se convierte en una primitiva accesible para cualquier área del negocio.

Esta misma página es un ejemplo del flujo: este sitio se generó, desplegó y publicó en https://docs.mystoredigital.cloud automáticamente, en una sola sesión, sin intervención manual entre la solicitud y la URL viva.
